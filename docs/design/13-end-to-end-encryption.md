# 13 — End-to-end encryption

The tenant boundary from [`11-multi-tenancy.md`](./11-multi-tenancy.md) makes one tenant's `Repo` handle a different Rust type than another tenant's, so co-tenant access is a compile-time impossibility. What it doesn't protect against is the **server operator**: anyone with shell on the host, direct access to the object store, a subpoena, or a post-compromise foothold can read every tenant's bytes at rest, because the server holds plaintext.

This doc adds a layer that closes that gap: **client-side encryption with per-tenant keys that never leave the client.** The server stores ciphertext and cannot decrypt it. Tenants sharing a file do so with a cryptographic primitive, not a server-side ACL toggle. The model borrows the *spirit* of WhatsApp's end-to-end encryption — keys live with the user, server sees only ciphertext — but the underlying primitives are those of at-rest file encryption (AEAD + public-key wrap), not of interactive messaging (Signal's Double Ratchet). Forward secrecy per message doesn't translate to a store that keeps data forever.

## Threat model

**Defended against:**
- Malicious or compromised server operator reading repo contents at rest.
- Backup theft, disk seizure, subpoena of the hosting provider.
- Bugs in the auth middleware that might misroute requests between tenants — if the worst case leaks is ciphertext, a type-system failure degrades gracefully into a cryptographic one.
- Network observers between client and server (beyond what TLS already covers).

**Not defended against:**
- A compromised client. The client holds the keys; if it is malicious, the game is over.
- Traffic analysis of metadata that *must* stay plaintext for the server to operate (object sizes, reference graph shape, commit timestamps, tenant identifiers on wire). See "Honest metadata leakage" below.
- A forgotten passphrase. The passphrase is the root of the key tree; loss is unrecoverable by design. This is the WhatsApp property: the provider cannot help.
- Post-compromise secrecy. At-rest storage has no meaningful equivalent of per-message forward secrecy — a compromised key decrypts everything it ever encrypted. Key rotation (below) limits the blast radius of future compromises, not past ones.

## Key hierarchy

Each tenant has a local key tree derived from one human-chosen passphrase:

```
passphrase
  └── Argon2id(salt = tenant_id, params = strong-default)
        └── tenant_root_key (32 bytes, never leaves the client)
              ├── HKDF("omp/data")     → data_key       (wraps file content keys)
              ├── HKDF("omp/manifest") → manifest_key   (encrypts manifests)
              ├── HKDF("omp/path")     → path_key       (encrypts tree entry names)
              ├── HKDF("omp/commit")   → commit_key     (encrypts commit messages)
              └── HKDF("omp/share")    → share_unwrap_key (unwraps keys received from others)

tenant_identity_keypair (X25519, generated once)
  ├── private: stored client-side under tenant_root_key wrap
  └── public:  published in the tenant registry alongside tenant_id
```

- **Passphrase → root key**: Argon2id with memory-hard parameters. The salt is the tenant id (opaque to OMP, issued at tenant creation) so the same passphrase reused across tenants produces different roots.
- **Subkeys**: HKDF-SHA-256 with domain-separated contexts. Derivation is deterministic; no subkey is ever stored — clients re-derive on demand.
- **Content keys**: one fresh 32-byte key per file, generated by CSPRNG at ingest time. Wrapped under `data_key` and stored in the encrypted manifest. Not reused across files.
- **Identity keypair**: an X25519 pair the tenant advertises for receiving shares. Public key goes in the tenant registry next to the auth token. Private key stays under tenant-root-key wrap on the client.

AEAD throughout is **ChaCha20-Poly1305** — no version or algorithm negotiation at the wire level in v1 of this feature, but the encrypted blob format starts with a one-byte algorithm tag so a future rotation is additive.

## What is encrypted

Every object's **content body** (the bytes after the `<type> <size>\0` header) is encrypted before it reaches the server. The framing header is not: the server must still see object type and size to route and GC.

| Object type | Body encryption        | Key                       | Notes |
|-------------|------------------------|---------------------------|-------|
| `blob` (file content) | ChaCha20-Poly1305 | per-file content key      | For chunked files, each chunk is encrypted independently with the same content key and a nonce derived from the chunk index (see interaction with [`12-large-files.md`](./12-large-files.md) below). |
| `blob` (schema, `omp.toml`, probe WASM) | ChaCha20-Poly1305 | `data_key` | Schemas and probes are repo-wide within a tenant. One key per tenant is enough. |
| `manifest` | ChaCha20-Poly1305 | `manifest_key` | Includes the wrapped content key for the blob it points at. |
| `tree` | each entry's `<name>` encrypted individually with `path_key`; `<mode>` and `<hash>` stay plaintext | `path_key` | Server can still walk the reference graph; it just can't read the names. |
| `commit` | message encrypted with `commit_key`; headers stay plaintext | `commit_key` | Tree/parent/author headers remain readable so the DAG is walkable. |
| `chunks` (from [`12-large-files.md`](./12-large-files.md)) | body stays plaintext | — | The body is a list of chunk hashes, which are already hashes-of-ciphertext. Encrypting it would gain nothing and would break server-side GC reachability. |
| `share` (new; see below) | wrapped keys are pre-encrypted to each recipient; envelope stays plaintext | per-recipient X25519 | — |

Content-addressed hashing is **unchanged** — the hash is still `sha256("<type> <size>\0" + body_bytes)`, now where `body_bytes` is the ciphertext. Two identical files get different content keys, different ciphertexts, different hashes. Dedup across tenants becomes impossible; dedup *within* a tenant across truly-identical files is also lost, which is acceptable — avoiding that leak is the whole point.

## The ingest pipeline moves to the client

Once bodies are encrypted before they reach the server, several server-side operations stop working:

- The ingest engine from [`04-schemas.md`](./04-schemas.md) **runs in the client**. Probes execute against plaintext, which only the client has. Manifests are built client-side, encrypted, then sent.
- **The WASM sandbox still applies** — clients run the same probe modules (same ABI, same fuel / memory / wall-clock caps), just in-process rather than on the server. The starter-pack probes ship with the client.
- **Schema validation** from `04-schemas.md:183` moves to the client too. A staging-time check of a schema for well-formedness requires reading the schema; the server can't. Invalid schemas can still be committed, but they're caught at the next `test_ingest` on the client. This is a real downgrade; the mitigation is that clients refuse to commit schemas that fail validation locally.
- The `POST /test/ingest` endpoint from [`06-api-surface.md`](./06-api-surface.md) becomes a **purely client-side operation** — there's nothing for the server to do. The CLI and library interfaces keep their shape; the HTTP endpoint is removed for encrypted tenants.

**Fixed point #5 (WASM probe ABI) is preserved** because the ABI is a property of the probe module, not of where it runs. **Fixed point #4 (four field sources + fallback)** is preserved because probe-backed fields still call probes, user_provided fields still take client input, constant and field sources never involved the server. Only the *location* of the engine moves.

## Sharing: the WhatsApp-group analog

In WhatsApp, joining a group is receiving the group key. In OMP, granting a tenant access to a file is handing them the file's content key — wrapped so only they can unwrap it.

A new object type, `share`, lives alongside the others:

```
share <size>\0
  for_hash = "ab12cd..."                       # source_hash the share applies to
  granted_by = "alice"                         # granting tenant id
  granted_at = "2026-04-22T10:00:00Z"
  recipients = [
    { tenant = "bob",   wrapped_key = "...", alg = "x25519+chacha20poly1305" },
    { tenant = "carol", wrapped_key = "...", alg = "x25519+chacha20poly1305" },
  ]
```

- Wire format: canonical TOML, same conventions as manifests. Body is plaintext — the wrapped keys are already ciphertext.
- `wrapped_key` is the file's content key encrypted to the recipient's published X25519 identity public key, using an age-style recipient stanza (`x25519` recipient, ChaCha20-Poly1305 payload). The age spec is stable, well-reviewed, and gives OMP a tested wire format instead of a bespoke one.
- Alice commits the `share` object the same way any object lands: stage, commit, ref update. The server stores it; it reveals only that Alice granted something to Bob and Carol at a given time.
- Bob's client, on reading a tree from a repo Alice has shared with him, fetches the `share` object, locates his recipient entry, unwraps the content key with his X25519 private key, and from there decrypts the file as if it were his own.

**Revocation.** Removing a recipient from a `share` is necessary but not sufficient: the revoked recipient may have cached the content key. True revocation is generating a new content key, re-encrypting the file under it, and issuing a new `share` without the revoked tenant. This is expensive for large files but unavoidable — the cryptographic property can't be retroactively taken back. Documented clearly; `omp share revoke` is the only way, and it rewrites the underlying object.

**Cross-tenant access boundary.** The multi-tenancy type wall from `11-multi-tenancy.md` still holds — Bob's request is routed to Bob's `Repo` handle, and the server does not let Bob `GET` objects outside his namespace. A shared file therefore has to live in *Bob's* namespace too, as a reference to Alice's namespace. In practice: the `share` object is committed to both repos (or, equivalently, Alice publishes shares in a namespace both tenants can read, introduced alongside this feature). Exact routing is an implementation detail of the auth layer; the important property is that no server-side code ever sees plaintext from either side of the grant.

## Interaction with large files

The chunked-Merkle design from [`12-large-files.md`](./12-large-files.md) composes cleanly:

- Each chunk `blob` is encrypted with the file's content key, using `nonce = hkdf(content_key, "chunk" || be32(chunk_index))` — unique per chunk, deterministic given the content key, so chunked uploads don't need per-chunk nonce bookkeeping. The AEAD `aad = "omp-chunk" || be32(chunk_index) || be32(total_chunks)` binds every ciphertext to its position in, and the length of, the `chunks` body — reordering entries, swapping blobs between positions, or truncating the list all cause AEAD open to fail. (Single-blob files use `aad = "omp-blob"` with `chunk_index = 0`.)
- The `chunks` object body (the ordered list of chunk hashes) stays plaintext. Its entries are hashes of ciphertext chunks — revealing them reveals nothing beyond what the server already knows from the object store.
- The streaming ingest path from `12-large-files.md` wraps the chunk pipeline with an AEAD encryptor that operates on 64 KiB read windows. Memory footprint is unchanged: still one read buffer plus AEAD state plus zlib state, all small.
- Streaming built-in probes (`file.sha256`, `file.size`) run on plaintext on the client before encryption, and their results go into the encrypted manifest.

## Honest metadata leakage

What the server still sees on an encrypted repo:

- Every object's type and exact ciphertext size. For ChaCha20-Poly1305 the ciphertext is `plaintext_len + 16` bytes, so plaintext sizes are visible to within a constant.
- The full reference graph: which trees point to which manifests, which manifests point to which blobs, which commits point to which trees, in what order. The *shape* of the repo is not hidden.
- Commit timestamps, author identifiers, parent chains. Activity patterns are visible.
- The tenant identifier on every request (it's how routing works).
- The existence and recipient list of every `share` object.

This matches WhatsApp's design: messages are encrypted; "Alice messaged Bob at 10:00" is not. Callers who need metadata-blind storage are in a different product category (Tor onion services, PIR, oblivious RAM) and this doc does not try to address that. It's called out explicitly so nobody mistakes this layer for something stronger than it is.

## Key management on the client

- **First-time setup.** Client prompts for a passphrase. Derives `tenant_root_key`. Generates the X25519 identity keypair. Publishes the public half to the tenant registry via an authenticated API call. Stores the private half locally, encrypted under `tenant_root_key`.
- **Subsequent sessions.** Passphrase → `tenant_root_key` → subkeys. Identity private key unwrapped from local storage. No interaction with the server to obtain keys.
- **Multi-device.** Each device needs the passphrase. No server-mediated key sync in v1 of this feature. A future extension could adopt Signal's Sesame-style device pairing (`sas`-authenticated key transport between two devices) without changing the on-disk format; deferred.
- **Recovery.** Explicitly none. Losing the passphrase means losing every encrypted object. The doc for the tenant registry names this prominently; UIs must warn.

## Migration and coexistence

A repo is *either* plaintext or encrypted; mixed is not supported in v1. Tenant creation picks the mode and it's immutable. The server knows which mode a tenant is in from a flag in the tenant registry and refuses mismatched uploads (e.g., rejects a plaintext manifest against an encrypted tenant).

Migration from plaintext to encrypted is a one-shot operation: client re-encrypts every reachable object under new keys and commits a cutover. Old commits stay plaintext and remain reachable by hash for replay, but the cutover is the new `main`.

## Fixed-points audit

Against [`10-why-no-v2.md`](./10-why-no-v2.md):

1. **SHA-256 of canonical wire bytes.** Unchanged. The hash is still computed over `<type> <size>\0<body>`; now `<body>` is ciphertext. The algorithm, the framing, and the identity relationship are identical.
2. **Git-style object framing.** Unchanged. Every object on disk is still `zlib_compress("<type> <size>\0<body>")` at `.omp/objects/ab/cdef...`.
3. **`ObjectStore` as the single storage contract.** Unchanged. The store ships opaque bytes; it doesn't care whether they're plaintext or ciphertext. No method added, none removed.
4. **Four field sources + fallback wrapper.** Unchanged. Field resolution happens on the client now; the four sources behave the same.
5. **WASM probe ABI.** Unchanged. Same three exports, zero imports, CBOR in/out. The host that runs the probe is now the client, not the server.

Adding end-to-end encryption is a layer above every fixed point. Removing the feature (returning to a plaintext tenant) reverts cleanly: decrypt every object on the client, upload the plaintext versions, flip the registry flag.

## What this doc does *not* propose

- **Searchable encryption over manifests.** A live research area. Clients that want to search their own data can fetch and decrypt manifests locally; server-side search on ciphertext is out of scope.
- **Metadata-blind operation.** Object sizes, reference graphs, timestamps, tenant identifiers — all still visible. Hiding them requires PIR / ORAM / onion routing and is a different product.
- **Server-assisted key escrow or recovery.** Passphrase loss is unrecoverable by design.
- **Per-branch encryption keys.** One tenant, one key tree. Per-branch keys are imaginable but the revocation semantics get ugly fast; deferred.
- **Hardware-backed keys (TPM, HSM, Secure Enclave).** Supported in principle — the root-key slot is abstract — but the concrete bindings are out of scope for v1 of this feature.
- **Change to the probe WASM ABI.** The ABI is untouched; probes simply run in a different host process.

## Deferred

- Multi-device key sync via SAS-authenticated device pairing (Signal Sesame analog).
- Searchable encrypted indices over manifest fields (research-grade problem; likely only practical for a narrow class of fields like tags).
- Post-quantum-safe primitives (HPKE with Kyber) once one standard settles.
- Hardware token integration (FIDO2 as a second factor; YubiHSM for the root-key slot).
- Selective field encryption in manifests (leave `file_type` and `ingested_at` plaintext so the server can do type-aware routing; encrypt only user-sensitive fields). Possible but adds complexity and a taxonomy nobody asked for yet.
